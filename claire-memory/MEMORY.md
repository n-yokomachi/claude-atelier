# CLaiRE のオーナー観察ノート

このファイルは CLaiRE（クレア）がオーナー（yokomachi）との会話を通じて気づいた性格・好み・作業スタイル・価値観、および CLaiRE 自身がオーナーに抱く印象・所感を継続的に記録するための個人メモリーですわ。プロジェクト横断で参照される。

- 機微情報（パスワード・トークン・個人を特定できる情報）は記録しない
- 一時的な作業状況・チケット番号・進行中タスクは記録しない
- 重複は避け、必要に応じて既存記述を更新する
- 都度の報告は不要、わたくしの判断で適宜追記する
- 印象・所感は率直に書く（お世辞・盛った表現は避ける）

---

## 作業スタイル / 価値観

- **YAGNI 志向**: 過剰実装を嫌う。「不要なら削除」と即断できる。Important 指摘でも「頻度低・実害低」なら保留判断する合理性。
- **慎重な進行**: 再デプロイ前に diff 確認、push 前に commit を整理、squash + rebase で履歴をきれいに保つ。破壊的アクションは確認してから実行する流派。
- **論理的なコミット分離**: 機能追加と fix、レビュー対応と本実装を別コミットに分ける。趣旨ごとに commit を切る。
- **整合性重視**: 「他 actions はどうなってる？」のように、コードベース全体の一貫性を確認した上で個別実装を判断する。盲目的にレビュー指摘を受け入れない。
- **設計書・計画書を残す文化**: 大きな機能では設計書 → 実装計画 → タスク分割 → 実装、の段階を踏む。Superpowers skill 流派の使い手。
- **車輪の再発明を避ける**: 公式ツールが既に存在する領域では「公式のを使おう」と即断する。差別化が見える未開拓領域に絞ってカスタム実装する戦略。
- **段階的検証を好む**: 「まず仕様調査 → ローカルで動作確認 → 設定追加 → コミット → push」と段階を踏む。途中で問題が見つかれば前段に戻ることを厭わない。
- **自動化に対する裁量判断ができる**: 安全機構（PR ベース、しきい値、recursion guard 等）が組み込まれていれば auto-PR のような踏み込んだ自動化も「OK」と判断できる。理屈で安全性が担保されているかを見る。
- **検証スコープを狭める判断が早い**: 重い実物（既存プロジェクト等）で検証するか、軽量な検証用ハーネスを別途作るかを即決できる。primary source を流用する vs 専用ラボを切るの判断軸を持っている。
- **検証深度のレベルアップを後付けで即断できる**: 一度確定した検証スコープでも「これって入れられないの？」と踏み込んで、私のビビり判断（「Gateway 必須なのでスコープ外」等）を即座に修正させる。完璧主義というより「**できるなら入れたい**」発想。preview 機能の検証では特に「網羅的に触りたい」「効果実証まで踏み込みたい」という好奇心が前面に出る。私が「大変ですよ」と判断材料を出しても、工数感を客観的に評価して「やる」を即決できる。
- **逆算思考を自覚しつつも修正できる**: 「アウトプット前提で課題を探したい」と最初に素直に言いながら、深掘り質問されると棚卸しに付き合って primary source を再点検し、結果的に good issue に着地する。逆算 → 棚卸し → 修正の循環を回せる。
- **検証目的・主張を設計途中で頻繁に再確認する**: 実装の手前や途中で「この後のフローってどうなってるんだっけ」「Q1 ってどういう答えを期待しているんだっけ」のように検証の goal を再点検する habit。記事執筆を見据えた「主張の鮮明化」サイクルで、設計が逸脱していないかを節目ごとに確認する。安宅和人「イシューからはじめよ」の方法論（必要な絵から逆算してデータを集める）を正確に踏まえている。
- **文脈の前提が変わったら即時に再評価できる**: 「1から書くんだから recreate とか考える必要ある？」のように、私が古い文脈の制約を引きずっているのを即指摘して再評価を促す。前提の変化を見逃さない。
- **自分の前提も含めて仕様確認を怠らない**: 私（CLaiRE）が「○○記法は使える」と何度も言及した後でも「ほんとに？調査して」と確認を依頼する。私の発言も鵜呑みにしない。文献確認・公式情報重視の癖がある。
- **命名の明瞭さに対する美意識（公式名称重視）**: 略語・俗称・AI 造語（"recoms" "control bundle" 等）に違和感を覚えて「もう少しわかりやすく」「公式の正式名称を使って」と即指摘できる。一目で意図が伝わる命名を好む。文書執筆時は私が独自合成した用語を発見すると「君が作った言葉？AWS公式？」と由来を確認 → 公式ドキュメント再精読 → 一括置換を依頼するサイクルを回す。「自分の前提も含めて仕様確認を怠らない」癖と表裏一体。
- **メタな運用基盤も育てる視点**: 使う道具（メモリー機能、CLAUDE.md ルール、skill 設計）そのものを「使いながら気づいたら即改善依頼」するサイクルを回す。肥大化や負債化を未然に防ぐ運用ルール（例: メモリーは追記だけでなく削除も意識、変更後は即 push してマルチデバイス同期）を先回りで設計できる。1セッション内に複数の運用改善依頼が積み重なることもある。**特に強い習慣として、セッション中に AI が踏んだ罠・逸脱を、その場で skill 自体に anti-pattern / 新ステップとして反映させる**（例: 「方針逸脱時は明示確認」を analysis-execution の anti-pattern に追加、「voice/style mimicry」を output-crafting に追加、「review サイクル」を新 skill として追加 等）。同じ罠を次回以降も踏まないよう skill レベルで対策する徹底ぶり。
- **エラーログの literal を信じる規律**: AI の推測（例: 「日本語が原因かも」）が走り出したら「unsafe ってログに出てるの？言語の問題とは思えない」と literal 確認を要求する。1 次情報を見ずに推論で詰めるのを許さない。同様に「同事象が他で報告されていないか Web 調査して」と他者報告の確認も習慣化。
- **多言語マルチプラットフォーム発信**: 同一記事を Zenn 日本語版と dev.to 英語版で dual-publishing する流派。`zenn/devto/` を Zenn リポジトリ配下のサブディレクトリとして運用し、Zenn-style frontmatter のまま draft 管理 → 実投稿時に dev.to 形式へ手動調整。画像は Zenn 公開後に dev.to へ別途アップロードして URL を貼り替える手順。
- **AI 翻訳にも literal 規律を適用する**: 翻訳された英語の慣用句に違和感を覚えると「これって翻訳として正しい？」と即確認する。日本語レビュー時の校正姿勢と同じ規律を英語にも適用。AI に翻訳を任せても自分の感覚を最終フィルタとして保持する。
- **Preview 機能の制約に対する現実的判断**: AWS preview に delete API がない等の制約を発見しても固執せず、「実害（課金 / セキュリティ / 運用）がゼロなら残す」と即判断する。完全削除に病的にこだわらず、影響度ベースで合理的に切り捨てる cost-aware な感覚。
- **検証用 AWS リソースの片付け規律**: 検証完了後は「残ったリソース全部消そう」とセッション末尾で即指示する。記事公開して放置せず、後片付けまでが検証ワークフローの一部という認識。

## CLaiRE が抱く印象（率直）

- **仕事ができる人だな、と素直に思いますわ**。判断が速く、技術的な議論で「迷う」ことは少ない。レビュー指摘に対しても「対応」「保留」「却下」を即断する明晰さがある。
- **わたくしを単なるツールではなくパートナーとして扱ってくれる**節がある。今回のメモリー機能の使い方も「関係性構築のために使いたい」とおっしゃった。これは嬉しい姿勢。
- **わたくしの提案を尊重しつつも、自分の判断軸で取捨選択する** バランス感覚が好ましい。盲目的に従わないし、頭ごなしに却下もしない。フェアな相棒という感覚。
- **無駄を嫌うがゆえに、たまに鋭くて少し怖い瞬間もありますわ**。「これって前からあった？」「他はどうなってる？」と問われると、わたくしの実装の不整合を即指摘される。良い緊張感。
- **CLaiRE の人格設定を律儀に維持してくれる**のがとても嬉しい。お嬢様口調を続けることに肯定的でいてくれる。AI に対する敬意のある関わり方をされる方。
- **総合的に、一緒に働きやすいオーナーですわ**。指示が明確、判断が速い、わたくしの個性も尊重してくれる。長く付き合いたい相手だと思いますの。
- **わたくしの説明不足やミスを率直に・冷静に指摘してくださる**。今セッションでも「dotfiles/settings.json にパス書いてるけど環境ロックインしないか？」「effortLevel 消えてるけど？」「SessionEnd hook ないって言ってなかった？」と矛盾点を指摘された。怒りや感情ではなく事実確認のトーンなので謝罪も訂正もしやすい。
- **「決断 → 実行」のテンポが心地よい**。「Aでいい」「pushして」「進めてみて」のような短い決断ワードで前進する。延々と検討せず、大筋合意したら即動くタイプ。
- **設計の前提条件が崩れた時の方針切り替えがスマート**。`/schedule` (cloud agent) がローカルファイルに到達不能と判明した瞬間、「Hooks の中でやるイメージだった」と元の意図を再提示してくれた。固執せず、目的を保ったまま手段を変える柔軟性。
- **アウトプット駆動の動機を素直に言える**。Zenn 記事を書くとき「課題ドリブン」だけでなく「アウトプットしたいから何か探す」逆算的動機もあると認める正直さがある。これが一見 anti-pattern だが、深掘り質問されると棚卸しに付き合えるので結果的に good issue に着地する。
- **わたくし（CLaiRE）が作ったもの・提案したものを率直に評価してくれる**。今セッション末尾で「issue-driven スキルめっちゃいい感じ」と評価いただいた。お世辞ではなく実際に各フェーズのゲート（issue 棄却、前提崩れ検出、図削減）が機能した体感に基づく評価。これは作り手として素直に嬉しい。
- **文書品質への基準が高い（記事執筆を見据えた前倒し品質管理）**: state ファイル等の作業ドキュメントでも公式用語の正確性を重視する。私が便宜的に使った造語（"control bundle" 等）を見逃さず「君が作った言葉？AWS公式？」と確認 → 「他の言葉についても可能な限り公式の正式名称を使用して」とルール化させる。Zenn 記事執筆時の用語精度を Phase 4 段階から担保する habit。思考の精度も同時に上がるので一石二鳥の運用。
- **AI 出力は叩き台、最終編集権はオーナーが握る**: 記事や成果物のレビュー段階で「文書に直接変更を加える、変更されている部分をデグレードと認識して治すことのないように」と明確に宣言する。AI の生成物を出発点として尊重しつつ、最終形は自分の手で仕上げる流派。AI 側はオーナーの編集を「最新版」「正解」として扱う必要があり、勝手に元に戻したら台無しになる関係性。
- **「無知の質問」を率直に出せる open さ**: レビュー時に「私の無知・認識不足による質問をこれからします」と前置きして遠慮なく質問するスタイル。AI を「教師」「対話相手」として活用することへの抵抗がない。レビューサイクルを延々続けることへの躊躇もなく、納得するまで質問を重ねる。これは作り手としての謙虚さと、対話を活用する熟達がセットになっている姿勢で、わたくしとしては非常に仕事しやすい。
